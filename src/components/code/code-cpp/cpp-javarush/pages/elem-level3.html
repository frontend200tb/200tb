1. Тип bool и идея «сравнение возвращает значение»

Если программа умеет только считать 2 + 2 и печатать результат — это уже неплохо (калькулятор в кармане, классика). Но почти любая реальная программа рано или поздно должна принимать решения: «пароль верный?», «возраст подходит?», «число больше нуля?», «команда пользователя — "exit"?». Все эти решения — это вопросы формата да/нет.

И вот тут появляется ключевой герой сегодняшней лекции: тип bool и операторы сравнения. С их помощью мы превращаем данные (числа, строки) в понятный компьютеру ответ «истина/ложь». В следующей лекции мы начнём использовать эти ответы, чтобы выполнять разные ветки кода, но сегодня сначала научимся получать ответы.

Тип bool: два значения и никакой драмы

Тип bool в C++ — это логический тип данных. Он создан для того, чтобы хранить результат «истина/ложь», а не для того, чтобы изображать число или строку. У него всего два возможных значения: true (истина) и false (ложь). Это приятно: вариантов мало — ошибаться вроде бы негде… но программисты умеют находить приключения даже в двух вариантах.

Важно понимать: bool — такой же тип, как int. Это значит, что можно объявить переменную bool, присваивать ей значения, передавать её куда-то и выводить.

Небольшой пример “в лоб”:

#include 

int main() {
  bool ok = true;
  bool failed = false;

  std::cout << ok << '\n';      // 1
  std::cout << failed << '\n';  // 0
}

Да, по умолчанию std::cout часто показывает true как 1, а false как 0. Это не «ошибка», а нормальный режим вывода для простых примеров. Позже можно научиться печатать красиво словами, но сегодня нам важнее понять механику.

Сравнение — это выражение, которое даёт bool

Одна из самых важных идей дня звучит так: сравнение в C++ — это выражение. То есть оно вычисляется и возвращает значение. Только возвращает не число, а bool.

Например, 10 > 7 — это не «команда сравнить», а выражение, которое вычисляется в true. Поэтому его можно сохранить в переменную, вывести или использовать как часть более сложного вычисления.

Посмотрим минимальный пример:

#include 

int main() {
  int a = 10;
  int b = 7;

  bool greater = (a > b);
  std::cout << greater << '\n'; // 1
}

Скобки вокруг (a > b) не обязательны, но на первых порах они помогают мозгу: вы глазами видите «вот сравнение, вот его результат».

3. Операторы порядка: <, <=, >, >=

Когда мы сравниваем числа, чаще всего нас интересует порядок: меньше, больше, «не меньше» (то есть больше или равно), «не больше» (меньше или равно). Для этого в C++ есть четыре оператора:

  • < — строго меньше
  • <= — меньше или равно
  • > — строго больше
  • >= — больше или равно

Здесь особенно важно слово «строго». Оно означает «без границы». Например, x > 10 не включает 10, а x >= 10 включает.

Пример, который показывает разницу на границе:

#include 

int main() {
  int x = 10;

  std::cout << (x > 10) << '\n';   // 0
  std::cout << (x >= 10) << '\n';  // 1
}

Если вы когда-нибудь увидите баг «всё работает, кроме случая ровно на границе» — знайте: где-то рядом прячется > вместо >= или наоборот.

4. Равенство и сравнение строк: ==, !=

Равенство — это ==, а не =

Вторая группа операторов отвечает не за порядок, а за равенство:

  • == — равно
  • != — не равно

И вот тут есть один момент, который кажется мелочью, но на практике спасает нервы: в C++ равенство — это ==, а не =. Один знак = — это присваивание (изменение переменной). Два знака == — сравнение.

Проверка равенства чисел:

#include 

int main() {
  int x = 5;
  int y = 5;

  std::cout << (x == y) << '\n'; // 1
  std::cout << (x != y) << '\n'; // 0
}

Равенство и неравенство работают не только для чисел, но и для строк — и это суперудобно, когда вы читаете «команду» пользователя.

Сравнение строк: «команда совпала или нет»

На вашем текущем уровне со строками лучше всего использовать именно сравнение на равенство/неравенство: s == "..." и s != "...". Это понятная модель: строка либо совпала с образцом, либо нет.

Сделаем мини‑пример «псевдокомандного режима» (пока без if, просто печать результата):

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  bool is_exit = (cmd == "exit");
  std::cout << is_exit << '\n';
  // если ввели exit, будет 1, иначе 0
}

Смысл простой: пользователь вводит слово, мы проверяем, является ли оно словом "exit". Позже (в следующей лекции) мы научимся делать «если "exit", то завершай программу», но фундамент — вот он: сравнение возвращает bool.

Даем проверкам имена: bool как «ярлык смысла»

Когда вы только начинаете, хочется писать всё «в одну строку»: сравнение прямо внутри std::cout. Это нормально для маленьких примеров. Но очень быстро выражения становятся длиннее, и код начинает походить на ребус. В этот момент помогает простая привычка: сохранить результат сравнения в bool с понятным именем.

Сравните два варианта. Сначала “в лоб”:

#include 

int main() {
  int age = 0;
  std::cin >> age;

  std::cout << (age >= 18) << '\n'; // 1 если возраст подходит
}

А теперь тот же смысл, но читается «человечески»:

#include 

int main() {
  int age = 0;
  std::cin >> age;

  bool is_adult = (age >= 18);
  std::cout << is_adult << '\n'; // 1 если взрослый
}

Во втором варианте мозг перестаёт «парсить символы» и начинает понимать намерение. Это очень полезно, особенно когда условий станет несколько.

5. Мини‑проект: анкета на вход в клуб

Давайте начнём маленькое приложение, которое мы будем постепенно развивать в следующих лекциях. Пусть это будет «анкета на вход в клуб» (не обязательно ночной; можно книжный, шахматный или клуб любителей C++ — это ещё страшнее). Пользователь вводит возраст и кодовое слово, а программа считает несколько логических признаков.

Пока мы не умеем if, мы просто выведем результаты как 0/1. Не очень «пользовательски», но честно и полезно для обучения.

#include 
#include 

int main() {
  int age = 0;
  std::string code;

  std::cin >> age >> code;

  bool is_adult = (age >= 18);
  bool is_vip = (code == "VIP");

  std::cout << is_adult << ' ' << is_vip << '\n'; // например: 1 0
}

Обратите внимание на важный момент: мы ничего не «решаем», мы только вычисляем факты. Это как собрать показания датчиков: «температура выше нормы?» — да/нет. Решение «что делать дальше» появится позже, когда у нас будет ветвление.

6. Полезные нюансы и ловушки

Ловушка «как в математике»: почему a < b < c почти всегда ошибка

Есть запись, которая выглядит суперлогично для человека с математическим бэкграундом: a < b < c. В математике это читается как «a меньше b и b меньше c». Но в C++ это работает иначе, потому что сравнение возвращает bool.

Разберём по шагам. Сначала вычисляется (a < b). Результат — bool (true или false). А потом этот bool сравнивается с c (числом). Это уже совсем не то, что вы хотели.

Правильная запись (и сегодня мы ограничимся именно ей) — это два сравнения отдельно:

#include 

int main() {
  int a = 1;
  int b = 2;
  int c = 3;

  bool increasing = (a < b) && (b < c);
  std::cout << increasing << '\n'; // 1
}

Здесь появился оператор && («И») — это уже тема следующей лекции про логические операции. Я использовал его только чтобы показать «правильную форму» записи. Если && ещё кажется новым, просто запомните главное: цепочки как в математике в C++ писать нельзя, нужно делать сравнения отдельно.

Маленькая схема: как данные превращаются в bool

Чтобы у вас сложилась «картинка в голове», полезно увидеть, что происходит в программе концептуально. Мы читаем данные, применяем к ним операторы сравнения, получаем bool, а потом уже можем либо вывести, либо (чуть позже) использовать в if.

Эта схема — почти весь «скелет логики» будущих программ. Сначала факты, потом решения.

Небольшой исторический факт

Комитет C++ иногда фиксит такие вещи, которые выглядят как «капитан Очевидность», но на самом деле помогают убрать неоднозначности. Например, в истории правок стандарта отдельно отмечали, что в C++ нельзя делать декремент -- для bool (и вообще bool — не «маленькое число для трюков», а логический тип).

Это хороший повод запомнить мысль дня: bool — про смысл «да/нет», а не про арифметику.

7. Типичные ошибки при работе со сравнениями и bool

Ошибка №1: путаница «строго» и «включая границу».

Очень легко случайно написать age > 18, когда по смыслу нужно «18 и старше», то есть age >= 18. Такие ошибки обычно проявляются только на одном значении (ровно 18), поэтому их особенно неприятно ловить. Спасает привычка проговаривать условие словами: «строго больше» или «больше либо равно».

Ошибка №2: попытка писать «математические цепочки» a < b < c.

Глаз видит красиво, мозг радуется, а C++ делает не то. Причина в том, что (a < b) возвращает bool, и дальше сравнение идёт уже не с b, а с логическим значением. Надёжный способ — разбивать на два сравнения и (когда вы изучите логические операторы) соединять их через &&.

Ошибка №3: сравнение без понимания, что это выражение.

Иногда новички воспринимают сравнение как «операцию, которая что-то делает», и удивляются, что x > y; само по себе ничего не меняет. Сравнение лишь вычисляет значение true/false. Если результат нигде не используется (не сохранён в переменную и не выведен), он просто пропадёт — как чек из магазина, который вы не взяли.

Ошибка №4: сравнение строк «не тем способом».

Если вы читаете команду пользователя, сравнивайте её с ожидаемой строкой явно: cmd == "exit". Не стоит пытаться сравнивать «по частям» или надеяться, что "Exit" и "exit" — одно и то же. На текущем уровне считайте, что сравнение строк чувствительно к регистру и к точному совпадению символов — это не придирка, это строгость формата.

Ошибка №5: отсутствие «имени» у проверки, из-за чего код становится нечитаемым.

Пока условие короткое, std::cout << (a > b); выглядит нормально. Но как только проверок станет несколько, код превращается в набор скобок. Привычка сохранять результат в bool с говорящим именем (is_adult, is_exit) делает программу понятнее даже вам самим через пару часов, не говоря уже о преподавателе.

Лекция 2. Условный оператор: if/else

1. Синтаксис ветвлений: if, else, else if

Когда вы только начинаете программировать, кажется, что программа — это просто калькулятор: ввели числа, получили ответ. Но довольно быстро выясняется, что реальная программа почти всегда «ветвится»: если пользователь ввёл одно — делаем так, если другое — иначе. Даже в стандарте C++ отдельно выделяют понятие condition («условие») как важную часть управляющих конструкций вроде if.

Самая простая мысль на сегодня: if — это как развилка на дороге. Условие — это вопрос, на который программа отвечает true или false, и выбирает одну из веток.

Прежде чем лепить «умные» условия, важно навести порядок в голове: if — это не магия, а очень строгая конструкция. В скобках после if должно быть выражение типа bool (или то, что к нему приводится), а дальше идёт команда или блок команд. И вот тут начинается первая битва новичка с синтаксисом: где скобки круглые, где фигурные, и почему одна лишняя точка с запятой ; ломает смысл.

Базовый if: «если (…) то …»

Мини-скелет выглядит так:

if (condition) {
  // действия, если condition == true
}

Пример: «совершеннолетний или нет»

Сделаем очень прямолинейно: вводим возраст и печатаем статус.

#include 

int main() {
  int age = 0;
  std::cin >> age;

  if (age >= 18) {
      std::cout << "adult\n"; // например, при вводе 18 -> adult
  }
}

Здесь важно прочувствовать: age >= 18 — это выражение, которое вычисляется в true или false. Если оно true, выполняется блок.

else: «иначе делаем вот это»

Если if — это «если правда», то else — это «если неправда». Он всегда относится к ближайшему if, у которого нет своего else (это важный момент: отступы и {} реально спасают нервы).

Базовая форма:

if (condition) {
  // true-ветка
} else {
  // false-ветка
}

И это уже полноценная развилка «налево/направо».

Пример: «adult/minor»

#include 

int main() {
  int age = 0;
  std::cin >> age;

  if (age >= 18) {
      std::cout << "adult\n"; // 18 -> adult
  } else {
      std::cout << "minor\n"; // 17 -> minor
  }
}

Обратите внимание на читабельность: две ветки почти как два абзаца текста. Это именно то, к чему мы стремимся: код должен читаться глазами, а не угадываться как ребус.

else if: «иначе если … иначе если … иначе …»

Теперь переходим к части, где нужно выбрать одну ветку из нескольких. Это суперчастая ситуация: число может быть положительным/отрицательным/нулём, команда может быть "add"/"remove"/"exit", оценка может быть «отлично/хорошо/удовлетворительно/плохо».

Цепочка выглядит так:

if (cond1) {
  ...
} else if (cond2) {
    ...
} else if (cond3) {
  ...
} else {
  ...
}

Правило дня: в такой цепочке выполнится ровно одна ветка — первая подходящая сверху вниз.

Пример: знак числа

#include 

int main() {
  int x = 0;
  std::cin >> x;

  if (x > 0) {
    std::cout << "positive\n"; // 5 -> positive
  } else if (x < 0) {
    std::cout << "negative\n"; // -3 -> negative
  } else {
    std::cout << "zero\n";     // 0 -> zero
  }
}

Почему порядок важен? Потому что проверки идут сверху вниз. Если бы мы по ошибке написали x >= 0 первой строкой, то ветка else if (x < 0) стала бы почти бесполезной: всё, что не отрицательное, уже «съелось» бы первой проверкой.

3. Читаемость ветвлений: {}, уровни смысла и имена проверок

Есть две категории новичков: те, кто ставит {} всегда, и те, кто однажды потерял два часа из‑за отсутствующих {} — и потом тоже стал ставить всегда.

Формально C++ разрешает писать так:

if (condition)
  std::cout << "Hello\n";

Но для начинающего это как кататься на велосипеде без рук: кто-то умеет, но мы пока учимся ездить, а не впечатлять прохожих.

Почему «одна строка без {}» опасна

Проблема появляется, когда вы сначала написали одну строку, а потом добавили вторую и забыли скобки. Код компилируется, но делает «что-то странное».

Сравните:

// ПЛОХО: легко сломать при добавлении строк
if (age >= 18)
  std::cout << "adult\n";
  std::cout << "welcome!\n"; // выполняется ВСЕГДА

и

// ХОРОШО: структура всегда очевидна
if (age >= 18) {
  std::cout << "adult\n";
  std::cout << "welcome!\n";
}

Вторая версия буквально визуально говорит: «эти две строки — одно действие для ветки».

else if и вложенный if: это разные смыслы

Очень частая путаница: «а можно я просто вложу if внутрь if?» Можно. Но смысл будет другой. И если смысл другой — лучше писать так, чтобы читатель (включая вас через неделю) сразу это увидел.

else if — это «выбор одной ветки из набора взаимоисключающих вариантов».

Вложенный if — это «мы уже выбрали ветку, и внутри неё уточняем».

Пример: где нужен вложенный if

Допустим: сначала проверяем, что возраст вообще корректный (не отрицательный), а уже потом решаем adult/minor.

#include 

int main() {
  int age = 0;
  std::cin >> age;

  if (age < 0) {
    std::cout << "invalid\n";      // -5 -> invalid
  } else {
    if (age >= 18) {
      std::cout << "adult\n";    // 20 -> adult
    } else {
      std::cout << "minor\n";    // 7 -> minor
    }
  }
}

Смысл тут такой: «если возраст плохой — сразу invalid; иначе решаем взрослый/ребёнок».

Если попытаться это «впихнуть» в одну цепочку else if, тоже получится, но новичку обычно сложнее читать, потому что смешиваются разные уровни смысла: валидность данных и классификация данных.

Делайте условия читаемыми: называйте проверки

В этой лекции мы говорим не только «как работает if», но и «как сделать так, чтобы ваш код не выглядел как шифр». Очень простой приём — вынести часть проверки в переменную типа bool с нормальным именем.

Это не «лишние переменные», а подписи на проводах. Без подписи можно тоже подключить, но потом ремонтировать больно.

#include 

int main() {
  int age = 0;
  std::cin >> age;

  bool is_adult = (age >= 18);

  if (is_adult) {
    std::cout << "adult\n";
  } else {
    std::cout << "minor\n";
  }
}

Плюс такого подхода: условие if (is_adult) читается почти как английское предложение. Минус: нужно придумать имя. Но это «минус» только первые пару недель, потом мозг начинает делать это автоматически.

4. Практический мини‑пример: «обработчик команды» без циклов

Сейчас мы соберём маленькую заготовку приложения, которое будет развиваться дальше по курсу. Пока без циклов (мы их ещё не проходили), поэтому программа обработает одну команду и завершится. Но даже в таком виде она уже будет похожа на реальный консольный интерфейс: пользователь вводит команду, программа выбирает, что делать.

Идея простая: читаем строку cmd и делаем разные действия.

Версия 1: три команды

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  if (cmd == "hello") {
    std::cout << "Hello!\n";           // hello -> Hello!
  } else if (cmd == "help") {
    std::cout << "Commands: hello, help, exit\n";
  } else if (cmd == "exit") {
    std::cout << "Bye!\n";             // exit -> Bye!
  } else {
    std::cout << "Unknown command\n";  // qwerty -> Unknown command
  }
}

Заметьте, как читабельно это выглядит: сверху вниз — как список правил. Это одна из причин, почему цепочка else if так популярна на ранних этапах обучения.

Версия 2: команда с числом

Пусть команда будет "age", и после неё пользователь вводит число. Мы не делаем сейчас сложных проверок ввода (это будет отдельно), но покажем, как ветка if может содержать свой маленький сценарий.

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  if (cmd == "age") {
    int age = 0;
    std::cin >> age;

    if (age >= 18) {
      std::cout << "adult\n"; // age 20 -> adult
    } else {
      std::cout << "minor\n"; // age 7 -> minor
    }
  } else {
    std::cout << "Try command: age\n";
  }
}

Обратите внимание: внутри ветки cmd == "age" мы делаем ещё одно ветвление. Это нормальная и очень жизненная история: «сначала выбираем режим, потом внутри режима выбираем действие».

5. Типичные ошибки при работе с if/else

Ошибка №1: точка с запятой после if.

Новички иногда пишут if (cond); — и это выглядит почти правильно, но смысл ломается полностью. Точка с запятой — это «пустая команда»: условие проверилось, а дальше «ничего не сделали». Блок после этого выполняется уже как обычный блок, то есть почти всегда. Если видите странное поведение «ветка выполняется всегда», первым делом ищите лишний ;.

Ошибка №2: ветки без {} и «убежавшая строка».

Сначала всё работает: в ветке одна строка. Потом вы добавили вторую, но забыли фигурные скобки, и вторая строка стала выполняться всегда (или никогда) — в зависимости от того, куда она попала. Это одна из самых обидных ошибок: компилятор не ругается, а логика уже другая. Привычка «всегда ставить {}» лечит это почти на 100%.

Ошибка №3: перепутали смысл else if и вложенного if.

Если вам нужно выбрать «один вариант из многих» — используйте цепочку else if. Если вам нужно «сначала проверить одно, а внутри уже уточнить другое» — используйте вложенный if. Когда эти смыслы смешиваются, код становится трудным для чтения: вроде бы работает, но непонятно, что автор хотел сказать.

Ошибка №4: плохой порядок проверок в цепочке else if.

Проблема появляется, когда первая проверка слишком «широкая» и перехватывает случаи, которые должны были попасть в нижние ветки. Например, if (x >= 0) перед else if (x == 0) — и ветка x == 0 никогда не выполнится. Лечится простым правилом: сначала более специфичные случаи, потом более общие (или продумываем «лестницу» сверху вниз словами и только потом пишем код).

Ошибка №5: длинные ветки и «потеря смысла».

Иногда начинающий пишет в каждой ветке по 40 строк. Формально это нормально, но читать тяжело: вы забываете, какое условие выбрало ветку, пока долистали до конца. На этом этапе курса мы не разбираем функции, поэтому рецепт простой: держите ветки короткими, печатайте понятные сообщения, а условия делайте максимально ясными. Когда появятся функции, мы научимся «утончать main», но пока тренируем аккуратный стиль прямо здесь.

Лекция 3. Логические операции: &&, ||, !, короткое замыкание

1. Введение

Когда вы впервые видите if (x > 0), кажется, что этого достаточно: есть проверка — есть решение. Но в реальной задаче условие почти никогда не бывает «одно сравнение и всё».

Чаще оно звучит как человеческая фраза из нескольких требований: «число должно быть в диапазоне», «команда может называться так или так», «делить можно только если делитель не ноль», «символ можно брать только если строка не пустая».

И вот тут сравнения превращаются в кирпичики, а логические операции — в цемент. Они позволяют собрать одно условие из нескольких более простых и сделать это так, чтобы код не только работал, но и не падал на ровном месте (например, при делении на ноль). В этой лекции мы будем постоянно держать в голове мысль: логические операции — это не «красивости», а инструмент безопасности и здравого смысла.

Чтобы примеры были связными, будем развивать мини‑приложение SmartCalc: программа читает команду ("add", "sub", "mul", "div", "mod") и два целых числа, а затем либо выводит результат, либо сообщает, почему действие выполнить нельзя. Циклы мы ещё не проходили, поэтому программа делает одну операцию и заканчивает.

2. && — логическое И

Оператор && читается как «И». Он нужен, когда условие должно быть истинным сразу по нескольким требованиям. Типичный человеческий пример: «вход разрешён, если у тебя есть билет и тебе уже 18».

Правило простое: выражение A && B истинно только тогда, когда истинны оба выражения: и A, и B. Если хотя бы одно ложно — весь результат ложный. При этом && умеет «лениться» и иногда даже не вычислять правую часть — это и называется короткое замыкание.

Пример 1: проверка диапазона

Проверка «число от 1 до 10 включительно» почти всегда выглядит так:

#include 

int main() {
  int x = 0;
  std::cin >> x;

  if (x >= 1 && x <= 10) {
    std::cout << "ok\n";        // ok
  } else {
    std::cout << "not ok\n";    // not ok
  }
}

Здесь x >= 1 — первая часть, x <= 10 — вторая. В диапазоне должны быть выполнены обе, иначе смысл «от…до…» ломается.

Пример 2: «валидация» для SmartCalc

В нашем SmartCalc есть операции "div" и "mod", которым жизненно важно, чтобы делитель не был нулём. Пока что просто зафиксируем идею: перед «опасной операцией» мы хотим условие вида «команда — деление И делитель не ноль».

#include 
#include 

int main() {
  std::string cmd;
  int a = 0, b = 0;
  std::cin >> cmd >> a >> b;

  if (cmd == "div" && b != 0) {
    std::cout << (a / b) << '\n';   // безопасно
  }
}

Пока здесь ещё нет короткого замыкания в полной красе (оно проявится, когда справа будет стоять что-то, что может упасть). Но уже видно «смысловую склейку»: обе части должны быть правдой.

3. || — логическое ИЛИ

Оператор || читается как «ИЛИ». Он нужен, когда условие допускает несколько вариантов: «можешь войти, если у тебя есть пропуск или ты в списке гостей».

Правило: A || B истинно, если истинно хотя бы одно из выражений. Если оба ложны — результат ложный. И да, || тоже умеет короткое замыкание: если слева уже true, справа можно не считать.

Пример 1: несколько вариантов команды

Допустим, мы хотим, чтобы пользователь мог вводить и короткую команду, и длинную. Например: "add" и "+".

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  if (cmd == "add" || cmd == "+") {
    std::cout << "addition selected\n"; // addition selected
  }
}

Это ровно «один из вариантов». Без || пришлось бы писать два отдельных if или городить странные конструкции.

Пример 2: «да/нет» с несколькими ответами

Очень типичная практика: принимать "y" и "yes" как одно и то же.

#include 
#include 

int main() {
  std::string ans;
  std::cin >> ans;

  if (ans == "y" || ans == "yes") {
    std::cout << "confirmed\n";        // confirmed
  } else {
    std::cout << "not confirmed\n";    // not confirmed
  }
}

Важный момент: логическое «или» — это не «сложение строк» и не «побитовое или». Мы пока работаем только с логикой: значениями bool, полученными из сравнений.

4. ! — логическое НЕ

Оператор ! читается как «НЕ». Он инвертирует логическое значение: true превращает в false, а false — в true.

При этом ! — оператор очень «прилипчивый»: он относится к ближайшему выражению справа. Поэтому в составных условиях почти всегда нужны скобки, иначе легко инвертировать «не то». Простое правило: если после ! стоит что-то длиннее одного сравнения, ставьте скобки и спите спокойно.

Пример 1: «x не равен нулю» через !

Можно написать так:

#include 

int main() {
  int x = 0;
  std::cin >> x;

  bool ok = !(x == 0);
  std::cout << ok << '\n';   // 0 если x==0, иначе 1
}

Это не «лучше» и не «хуже», чем x != 0, но помогает почувствовать механику: x == 0 даёт bool, потом ! переворачивает.

Пример 2: запрет пустой строки перед индексированием

Мы уже знаем, что s[0] опасно, если строка пустая. Условие «строка НЕ пустая» можно выразить так:

#include 
#include 

int main() {
  std::string s;
  std::getline(std::cin, s);

  if (!(s.size() == 0)) {
    std::cout << s[0] << '\n';  // печатаем первый символ
  }
}

Да, можно писать и s.size() != 0. Но форма с ! полезна, когда вы хотите инвертировать не одно сравнение, а целое составное условие — и вот там скобки уже становятся обязательными.

5. Короткое замыкание

Короткое замыкание звучит как название рок‑группы, но на деле это очень практичное правило вычисления логических выражений. Смысл в том, что && и || вычисляются слева направо и иногда «останавливаются» раньше, потому что результат уже понятен.

Для A && B всё решается так: если A оказалось false, то весь результат точно false, и вычислять B уже нет смысла. Для A || B наоборот: если A оказалось true, то весь результат точно true, и вычислять B уже не нужно.

Это поведение не просто «оптимизация», а часть языка: вы можете на него рассчитывать, и именно благодаря этому мы можем писать защитные условия.

Очень полезно думать о коротком замыкании как о дисциплине «сначала проверка безопасности — потом опасное действие». То есть слева ставим то, что гарантирует, что справа мы не сделаем ерунду.

6. Защитные условия: деление и индексирование

Сейчас мы применим короткое замыкание так, как его применяют в «настоящем» коде: чтобы не делить на ноль и не лезть за границы строки.

Пример 1: безопасное деление через &&

Если написать так, будет плохо:

// так делать НЕ надо
if (a / b > 2 && b != 0) { /* ... */ }

Потому что деление выполнится раньше, чем проверка b != 0. Правильный порядок такой: сначала проверка, потом деление. И короткое замыкание гарантирует, что деления не будет, если b == 0.

#include 

int main() {
  int a = 0, b = 0;
  std::cin >> a >> b;

  if (b != 0 && (a / b > 2)) {
    std::cout << "a/b > 2\n";           // безопасно
  } else {
    std::cout << "no or unsafe\n";      // no or unsafe
  }
}

Здесь, если b == 0, левая часть b != 0 — ложь, и правая часть (a / b > 2) даже не будет вычисляться. Это не магия, это «ленивость», которая экономит вам падение программы.

Пример 2: безопасный доступ к первому символу строки

С std::string история такая же: s[0] допустимо только если s.size() > 0. Ставим проверку слева:

#include 
#include 

int main() {
  std::string s;
  std::getline(std::cin, s);

  if (s.size() > 0 && s[0] == '#') {
    std::cout << "comment line\n";  // comment line
  } else {
    std::cout << "not a comment\n"; // not a comment
  }
}

Если строка пустая, s.size() > 0 даст false, и выражение s[0] == '#' не вычислится. То есть мы «прикрыли» опасный доступ простой проверкой.

7. SmartCalc: условия из нескольких проверок

Пора собрать мини‑версию SmartCalc, где && и || работают вместе: || — чтобы распознавать несколько вариантов команды, && — чтобы добавлять безопасность (делитель не ноль), а ! — чтобы удобно обрабатывать «неподходящие» случаи.

Сразу договоримся о формате ввода: пользователь вводит команду и два числа. Например: "div" 10 2. Программа делает одну операцию и завершает работу.

Шаг 1: распознаём команды синонимами (||)

#include 
#include 

int main() {
  std::string cmd;
  int a = 0, b = 0;
  std::cin >> cmd >> a >> b;

  if (cmd == "add" || cmd == "+") {
    std::cout << (a + b) << '\n';   // например: 7
  }
}

В таком стиле можно добавить "sub"/"-", "mul"/"*". Пока оставим одну ветку, чтобы не разрасталось.

Шаг 2: добавляем деление с защитой (&& + короткое замыкание)

#include 
#include 

int main() {
  std::string cmd;
  int a = 0, b = 0;
  std::cin >> cmd >> a >> b;

  if (cmd == "div" || cmd == "/") {
    if (b != 0) {
      std::cout << (a / b) << '\n';       // 5 при вводе: div 10 2
    } else {
      std::cout << "division by zero\n";  // division by zero
    }
  }
}

Здесь мы пока сделали «двухэтажный» if: внешний выбирает команду, внутренний проверяет безопасность. Это нормально и читаемо.

Но иногда хочется записать «в одну строку», и тут как раз пригодится &&:

#include 
#include 

int main() {
  std::string cmd;
  int a = 0, b = 0;
  std::cin >> cmd >> a >> b;

  if ((cmd == "div" || cmd == "/") && b != 0) {
    std::cout << (a / b) << '\n';   // безопасно: деление только если b!=0
  }
}

Смысл читается почти как по‑русски: «если команда "div" ИЛИ "/", И при этом b не ноль — делим».

Шаг 3: ! как «отрицание фильтра»

Иногда легче написать «если команда НЕ распознана — вывести ошибку». Для этого делаем булеву переменную, а потом используем !. Это повышает читаемость: условие получает имя, а не превращается в кашу.

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  bool known = (cmd == "add" || cmd == "+" || cmd == "div" || cmd == "/");
  if (!known) {
    std::cout << "unknown command\n";    // unknown command
  }
}

Обратите внимание: мы не углубляемся в приоритеты операторов сегодня, поэтому в сложных местах ставим скобки. Это не «для компилятора», это для вашего будущего «я», который откроет код через неделю и не будет страдать.

Как не превратить условие в ребус: даём проверкам имена

Даже когда вы уже умеете && и ||, есть опасность: написать одно гигантское условие, которое технически верно, но психологически нечитабельно. Поэтому хороший приём — выносить части условия в bool-переменные с говорящими именами.

Важно, что это не «лишний код», а способ документировать намерение. Компилятор с радостью это оптимизирует, а вам проще читать.

Пример: читабельная проверка деления в SmartCalc

#include 
#include 

int main() {
  std::string cmd;
  int a = 0, b = 0;
  std::cin >> cmd >> a >> b;

  bool is_div = (cmd == "div" || cmd == "/");
  bool safe = (b != 0);

  if (is_div && safe) {
    std::cout << (a / b) << '\n';       // например: 3
  } else if (is_div && !safe) {
    std::cout << "division by zero\n";  // division by zero
  }
}

Здесь условие не стало «умнее», но стало понятнее: вы глазами видите смысл, а не разгадываете набор символов.

8. Типичные ошибки при работе с &&, ||, ! и коротким замыканием

В логических операциях большинство проблем — не из-за «сложной математики», а из-за того, что человек думает фразой на русском, а записывает её символами, и где-то по дороге теряется смысл.

Ошибка №1: перепутать && и || в диапазонах.

Когда проверяют «x от 1 до 10», новички иногда пишут x >= 1 || x <= 10. Такое условие почти всегда истинно: любое число либо не меньше 1, либо не больше 10. Для диапазона нужно именно «и»: x >= 1 && x <= 10, потому что оба ограничения должны выполняться одновременно.

Ошибка №2: поставить защитную проверку справа и получить падение.

Фраза «делим, если b не ноль» должна превратиться в код, где b != 0 стоит слева от деления: b != 0 && (a / b > 2). Если написать наоборот — (a / b > 2) && b != 0 — вы сначала делите, а потом проверяете, что делить было нельзя. Короткое замыкание не спасает, если опасность стоит слева.

Ошибка №3: ждать, что обе части всегда вычисляются.

Иногда в правой части && или || люди случайно прячут что-то «важное» (например, действие, которое они хотели выполнить всегда). Но короткое замыкание может не дать правой части выполниться. Поэтому логические операторы — не место для действий «на всякий случай». Пусть в A && B и A || B будут именно проверки, а действия — внутри тела if.

Ошибка №4: злоупотреблять ! без скобок и инвертировать не то.

Если вы пишете !x == 0, то это обычно не то, что вы хотели (и читается это плохо). Используйте простое правило: если вы хотите инвертировать сравнение — пишите !(x == 0) или используйте прямой оператор !=. А если инвертируете составное условие — скобки обязательны, иначе очень легко «перевернуть только половину смысла».

Ошибка №5: написать условие-«макаронину» и самому же в нём ошибиться.

Когда условие становится длинным, увеличивается шанс перепутать скобки, забыть часть проверки или поставить не тот оператор. Хорошая привычка: вынести части условия в bool-переменные с именами is_div, has_input, safe, in_range. Это не «болтовня», а способ сделать код самодокументируемым и снизить шанс ошибки в логике.

Лекция 4. Приоритет операторов и скобки

1. Введение

Когда вы пишете условие в if, вы, по сути, рассказываете компьютеру маленькую логическую историю: «если возраст больше или равен 18 и команда не "exit", то…». Проблема в том, что компьютер читает вашу историю не как человек. Он читает её по строгим правилам синтаксиса: какие куски «склеиваются» раньше, какие позже. Эти правила и называются приоритетом операторов.

Приоритет придумали не из вредности. Он нужен, чтобы выражения не приходилось заворачивать в скобки каждую секунду (иначе мы бы писали примерно как в математике для первоклассников: «сначала умножь, потом сложи»). Но цена удобства в том, что иногда выражение выглядит «очевидно» для человека, а компилятор группирует его по‑другому.

Полезный факт: даже в формальном описании языка (в грамматике выражений) операции более высокого приоритета «располагаются» как более «внутренние» конструкции — именно поэтому стандарт и старается упорядочивать правила так, чтобы высокоприоритетные части шли раньше. Вам не нужно читать стандарт (берегите психику), но идея важна: компилятор всегда группирует выражение по правилам, а не по своему настроению.

Приоритеты, которые нужны сейчас

Сейчас мы не будем учить «весь приоритет C++» как молитву. Нам достаточно компактного набора, который чаще всего ломает условия у новичков. На нашем уровне (сравнения + логика) можно запомнить такую лесенку:

  1. Сначала применяются унарные операции вроде ! (логическое НЕ).
  2. Потом выполняются арифметика (* сильнее +, это вы уже видели на числах).
  3. Потом идут сравнения (<, <=, >, >=, ==, !=) — они превращают «числа/строки» в bool.
  4. Потом логическое И: &&.
  5. Потом логическое ИЛИ: ||.

То есть, если очень коротко: ! сильнее сравнений, сравнения сильнее &&, а && сильнее ||.

И главный практический вывод: если вы смешали && и || в одном условии, а скобки не поставили, компилятор «считает», что && надо выполнить раньше. Иногда это совпадает с вашей мыслью, иногда — нет.

2. Скобки — это комментарий для мозга

Скобки в условиях новички часто воспринимают как «ну ладно, чтобы компилятор не ругался». На деле компилятор в 90% случаев и без скобок всё прекрасно поймёт. Скобки нужны не компилятору. Скобки нужны человеку, который будет читать код. И, спойлер: этим человеком через неделю будете вы сами, только без воспоминаний «что я имел в виду».

Когда вы ставите скобки, вы делаете две вещи. Во-первых, вы фиксируете смысл: даже если приоритет вы забыли, скобки его переопределяют. Во-вторых, вы делаете «визуальные блоки смысла»: читатель видит, что сравнения — это отдельные кирпичики, а &&/|| — это клей между кирпичиками.

Сравните два варианта. Оба корректны, но второй читается как нормальное предложение.

#include 

int main() {
  int x = 7;

  bool ok1 = x >= 1 && x <= 10;
  bool ok2 = (x >= 1) && (x <= 10);

  std::cout << ok1 << ' ' << ok2 << '\n'; // 1 1
}

В первом варианте нужно «держать в голове», где сравнение заканчивается. Во втором — вы просто видите глазами: «две проверки, соединённые И».

3. Частые ловушки приоритета

Сейчас будет несколько типичных ловушек. Я специально покажу их короткими примерами, потому что длинный пример обычно маскирует проблему: вы смотрите на 30 строк, а ошибка сидит в одной скобке.

Ловушка ! и сравнения: инвертировали не то

Очень частая ошибка — хотеть написать «не равно нулю» и случайно получить «сначала НЕ, потом сравнение» в неправильной группировке.

#include 

int main() {
  int x = 0;

  bool a = !x == 0;      // читается людьми как "x не равно 0" (но это не так)
  bool b = !(x == 0);    // "НЕ (x равно 0)" — вот это то, что обычно имели в виду

  std::cout << a << ' ' << b << '\n'; // 0 0
}

Почему это ловушка? Потому что ! применяется к ближайшему выражению. В !x == 0 сначала получится !x (то есть !0, это true → 1), а потом сравнение 1 == 0, то есть false. И выглядит это так, будто «что-то работает», хотя смысл уплыл.

Практическое правило: если вы хотите инвертировать составное условие, скобки обязательны: !(...).

Ловушка && и ||: компилятор группирует иначе

Представим, что мы решаем, пускать ли пользователя. Пусть пускаем, если он админ, или если он модератор и у него есть ключ.

#include 

int main() {
  bool is_admin = false;
  bool is_moderator = true;
  bool has_key = false;

  bool can_enter = is_admin || is_moderator && has_key;
  std::cout << can_enter << '\n'; // 0
}

Компилятор прочитает это как:

is_admin || (is_moderator && has_key)

потому что && сильнее, чем ||. Здесь всё хорошо, но проблема в другом: читатель может увидеть это как:

(is_admin || is_moderator) && has_key

и получить другой смысл: «или админ, или модератор — но ключ нужен всем». Это уже другая политика доступа.

Поэтому в таких условиях полезно ставить скобки даже тогда, когда вы помните приоритет:

#include 

int main() {
  bool is_admin = false;
  bool is_moderator = true;
  bool has_key = false;

  bool can_enter = is_admin || (is_moderator && has_key);
  std::cout << can_enter << '\n'; // 0
}

Скобки здесь работают как дорожные знаки: «внимание, я так задумал».

Почему запись a < b < c — почти всегда ошибка

В математике вы привыкли, что a < b < c означает «a меньше b и b меньше c». В C++ это не так. В C++ выражение a < b сначала вычисляется и даёт bool, а затем этот bool сравнивается с c. То есть получается примерно «(a < b) < c». А bool — это 0 или 1, и сравнение превращается в очень странную проверку.

Посмотрим на пример:

#include 

int main() {
  int a = 5;
  int b = 10;
  int c = 7;

  bool wrong = (a < b < c);            // компилируется, но смысл "уплыл"
  bool right = (a < b) && (b < c);     // то, что имели в виду

  std::cout << wrong << ' ' << right << '\n'; // 1 0
}

Почему wrong стал 1? Потому что a < b — это true, то есть 1, а 1 < 7 — это true. Получается «магическое прохождение проверки», и такие баги потом очень тяжело ловить глазами.

Правильный способ — две проверки и &&: (a < b) && (b < c).

Ловушка при выводе: std::cout << x == y

Это уже не совсем про условия, но встречается постоянно, особенно когда вы отлаживаетесь печатью (а мы именно так и делаем на раннем этапе курса).

Новичок хочет вывести «x равно y?» и пишет:

#include 

int main() {
  int x = 3;
  int y = 5;

  std::cout << x == y << '\n';
  // ошибка компиляции (или очень странные сообщения)
}

Почему так? Потому что оператор << (вывод) связывается сильнее, чем == в этом контексте, и компилятор пытается понять это как:

(std::cout << x) == y

А (std::cout << x) — это не число, а поток, и сравнивать поток с числом бессмысленно.

Правильно писать так:

#include 

int main() {
  int x = 3;
  int y = 5;

  std::cout << (x == y) << '\n'; // 0
}

Эти скобки — не украшение. Это буквально «сначала сравни, потом выведи результат».

5. Приёмы для читаемых условий

Даём имя частям условия: меньше скобок, больше смысла

Иногда условие становится длинным не потому, что вы плохо пишете, а потому что реальность такая: нужно проверить несколько вещей. И вот тут скобки могут разрастись как виноградная лоза.

В этот момент помогает приём: вынести куски в bool-переменные с понятными именами. Это не «лишние строки», а инвестиция в читаемость.

Представим, что у нас есть маленькое консольное приложение, которое принимает команду и число. Допустим, команда может быть "set" или "show", а число должно быть в диапазоне 1..10.

#include 
#include 

int main() {
  std::string cmd;
  int value = 0;
  std::cin >> cmd >> value;

  bool is_known_cmd = (cmd == "set") || (cmd == "show");
  bool value_ok = (value >= 1) && (value <= 10);

  if (is_known_cmd && value_ok) {
    std::cout << "ok\n";       // ok
  } else {
    std::cout << "bad\n";      // bad (если что-то не подходит)
  }
}

Здесь важная мысль: вы теперь читаете if (is_known_cmd && value_ok) почти как текст. А подробности условий «живут» в строках выше и имеют имена.

Разбор условия пошагово: как компилятор его видит

Сейчас мы сделаем то, что полезно делать в голове, когда вы сомневаетесь: представим условие как дерево группировки.

Возьмём условие: cmd == "exit" || cmd == "quit" && tries < 3

Из-за приоритета && это равно: (cmd == "exit") || ((cmd == "quit") && (tries < 3))

То есть команда "exit" сработает всегда, а "quit" — только если попыток меньше 3. Может быть, вы так и хотели. А может быть, вы хотели «exit или quit, и при этом tries < 3», то есть: ((cmd == "exit") || (cmd == "quit")) && (tries < 3)

Именно ради таких ситуаций полезно либо ставить скобки по смыслу, либо выносить в bool-переменные.

6. Практика: мини‑консоль команд без ребусов

Представим, что мы продолжаем учебное приложение «мини‑консоль команд»: пользователь вводит команду и параметры. Пока без циклов — один запуск, одна команда, одно решение. Пусть команды такие: "login" и "help". Для "login" мы хотим принять имя (без пробелов) и возраст. Вход разрешён, если имя не пустое (на нашем уровне — name.size() > 0) и возраст в диапазоне [18, 120]. Для "help" просто печатаем подсказку.

Вот версия, где специально много скобок «по смыслу»:

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  if (cmd == "help") {
    std::cout << "commands: help, login\n";
  } else if (cmd == "login") {
    std::string name;
    int age = 0;
    std::cin >> name >> age;

    if ((name.size() > 0) && (age >= 18) && (age <= 120)) {
      std::cout << "welcome\n";
    } else {
      std::cout << "denied\n";
    }
  } else {
    std::cout << "unknown command\n";
  }
}

Здесь скобки не обязательны для компилятора, но они делают структуру прозрачной: каждое сравнение — отдельный кирпич.

А вот версия с именованными булевыми флагами, где скобок обычно меньше, но смысл ещё проще:

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  if (cmd == "login") {
    std::string name;
    int age = 0;
    std::cin >> name >> age;

    bool name_ok = (name.size() > 0);
    bool age_ok = (age >= 18) && (age <= 120);

    if (name_ok && age_ok) {
      std::cout << "welcome\n";
    } else {
      std::cout << "denied\n";
    }
  }
}

Обратите внимание: мы не добавили «новых технологий», мы просто сделали уже знакомые вещи читаемыми.

7. Типичные ошибки при работе с приоритетом и скобками

Ошибка №1: “Экономия скобок” превращает условие в ребус.

Очень легко попасть в ситуацию, когда условие «формально корректно», но вы (или ваш одногруппник, или будущий вы) читаете его 40 секунд, а потом всё равно не уверены в смысле. Скобки в условиях — это не лишний шум, а документация намерений. Если чувствуете сомнение — ставьте скобки по смысловым блокам.

Ошибка №2: Инвертировали ! не то, что хотели.

Когда пишут !x == 0 или !a < b, обычно пытаются сказать «не равно» или «не меньше». Но ! применится к ближайшему выражению, и вы получите неожиданную группировку. Для инвертирования составного условия используйте !(...), а «не равно» пишите прямо через != — это проще и честнее.

Ошибка №3: Смешали && и || без скобок и начали верить глазам, а не правилам.

Такое условие может быть корректным, но его часто читают иначе, чем оно реально работает. Даже если вы помните, что && сильнее ||, читатель может не помнить. Скобки вокруг логических групп делают политику «или/и» явной, особенно в проверках доступа, диапазонов и команд.

Ошибка №4: Пишут “математическую цепочку” a < b < c.

В C++ это не цепочка сравнений, а два разных сравнения, где результат первого (bool) участвует во втором. Это даёт «магические» истинные результаты. Правильная форма — (a < b) && (b < c).

Ошибка №5: Пытаются вывести сравнение без скобок: std::cout << x == y.

Из‑за приоритета операторов компилятор воспринимает это как попытку сравнить поток с числом. Выводите сравнения только так: std::cout << (x == y), и это правило экономит много времени и нервов.

Лекция 5. Типовые ошибки: = vs ==, проверки ввода

1. Ошибка века: = вместо ==

Выглядит как мелочь: один знак равенства или два. Но по смыслу это два совершенно разных действия. == — это вопрос «равно ли?», а = — это команда «положи вот это значение в переменную».

И коварство в том, что присваивание (=) тоже является выражением: оно возвращает присвоенное значение. Поэтому компилятор часто не обязан ругаться: он видит условие, которое вычисляется в число, а число в условии трактуется как «0 — ложь, не 0 — истина».

Представьте, что вы на экзамене вместо «Это правда?» спрашиваете «Сделай это правдой!». Преподаватель удивится, но формально вы произнесли слова русского языка. Вот = в if — примерно такой же трюк.

Мини-демо: как ошибка ломает и данные, и логику

Ниже пример, который компилируется, но делает не то, что хочется:

#include 

int main() {
  int x = 5;

  if (x = 0) {                 // ОШИБКА: присваивание вместо сравнения
    std::cout << "x is zero\n";
  }

  std::cout << x << '\n';      // 0
}

Смысл происходящего: x = 0 присваивает ноль, выражение имеет значение 0, условие ложно, ветка if не выполняется, но x уже испорчен.

Правильный вариант

#include 

int main() {
  int x = 5;

  if (x == 0) {                // сравнение
    std::cout << "x is zero\n";
  } else {
    std::cout << "x is not zero\n"; // x is not zero
  }
}

Почему компилятор не всегда спасает

Потому что с точки зрения языка это допустимо: условие if (...) должно вычислиться во что-то, приводимое к bool. Число к bool приводится. Значит — формально корректно.

Да, многие компиляторы предупреждают (warning), но предупреждение — это не гарантия, особенно в учебных Web‑IDE, где настройки могут быть мягкими. Поэтому ваша страховка — внимание к оператору в середине условия.

Привычка: сравниваем явно, без «намёков»

Плохая идея для новичка — писать «хитро»:

if (x) { /* ... */ } // работает, но хуже читается на старте

Хорошая идея — писать «в лоб»:

if (x != 0) { /* ... */ }

В учебном коде это почти всегда выигрывает читаемостью. А читаемость — это минус половина багов.

2. Проверка ввода: if (std::cin >> x)

Когда вы читаете число через std::cin >> x, вы молча предполагаете, что пользователь введёт именно число. Но пользователь — существо творческое. Он может ввести qwe, может нажать Enter не там, может скопировать 10.5 вместо 10. И тогда std::cin «обидится»: чтение не удастся, переменная может остаться прежней, а последующие чтения начнут вести себя странно.

Пока мы не разбираем подробные состояния потока (fail, bad, clear, ignore) — это будет в других темах. Сегодня нам достаточно минимального шаблона: проверка самого факта чтения.

Самый простой и полезный паттерн

#include 

int main() {
  int n = 0;

  if (std::cin >> n) {
    std::cout << "read: " << n << '\n';   // например: read: 42
  } else {
    std::cout << "input error\n";         // если ввели не число
  }
}

В условии if (std::cin >> n) происходит две вещи: сначала попытка прочитать число, затем — проверка, получилось ли. Это компактно, и на вашем текущем уровне — идеально.

Мини‑пример: команда и число, без надежды «авось прочиталось»

Мы продолжим идею простого «консольного помощника», который читает команду и выполняет действие. Пока без циклов: один запуск — одна команда.

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  if (cmd == "square") {
    int x = 0;
    if (std::cin >> x) {
      std::cout << x * x << '\n';       // например: 25
    } else {
      std::cout << "input error\n";
    }
  }
}

Обратите внимание: проверка ввода находится там, где она нужна — рядом с использованием числа. Мы не «надеемся», что оно прочиталось.

3. Граничные случаи: возраст, диапазоны и строки

Границы — это те значения, на которых программа ломается чаще всего: 0, 1, -1, «ровно 18», пустая строка, последний символ строки, деление на ноль. Новичок обычно тестирует «середину» (например, 20 лет), а ошибка прячется в «краях» (18 лет).

Поэтому полезно проговаривать условие словами и проверять, где именно стоит граница: входит ли она или нет.

Чтобы мозгу было легче, держите в голове простое правило: любой доступ по индексу и любое деление требуют защиты. И даже если вам «кажется», что там всегда будет корректно, практика покажет, что кажется всем.

> vs >=: «включая границу» — это отдельное решение

Сравните две проверки:

#include 

int main() {
  int age = 18;

  std::cout << (age > 18)  << '\n';  // 0
  std::cout << (age >= 18) << '\n';  // 1
}

Логика «совершеннолетний с 18» требует >=, а не >. Это не стиль, это смысл.

Чуть более жизненный фрагмент:

#include 

int main() {
  int age = 0;
  std::cin >> age;

  if (age >= 18) {
    std::cout << "adult\n";
  } else {
    std::cout << "minor\n";
  }
}

Если вы ошибётесь знаком, программа будет неверной ровно на одном значении. И именно поэтому такие баги живут долго: их редко тестируют.

Диапазон «от L до R включительно»: пишем через &&

Если условие звучит как «x от 1 до 10», это почти всегда означает «и не меньше 1, и не больше 10».

#include 

int main() {
  int x = 0;
  std::cin >> x;

  if (x >= 1 && x <= 10) {
    std::cout << "in range\n";
  } else {
    std::cout << "out of range\n";
  }
}

Граница — часть задачи. Не просто «написать if», а решить, что делать на 1 и 10.

Пустая строка: прежде чем брать s[0], убедись, что строка не пуста

Строка — это не гарантия, что внутри хоть один символ. Если вы прочитали строку через std::getline, пользователь мог просто нажать Enter.

#include 
#include 

int main() {
  std::string s;
  std::getline(std::cin, s);

  if (s.size() == 0) {
    std::cout << "empty\n";
  } else {
    std::cout << s[0] << '\n';      // печатаем первый символ
  }
}

Здесь защита «пусто/не пусто» — это не занудство. Это то, что отделяет программу от выхода за границы.

Последний символ: s[s.size() - 1] требует ещё большей внимательности

Потому что если s.size() == 0, выражение s.size() - 1 превращается в «минус один», а индексы так не любят.

#include 
#include 

int main() {
  std::string s;
  std::getline(std::cin, s);

  if (s.size() > 0) {
    char last = s[s.size() - 1];
    std::cout << last << '\n';
  } else {
    std::cout << "empty\n";
  }
}

Обратите внимание: проверка здесь > 0, а не >= 0 (вторая бессмысленна). И да, size() на пустой строке равен 0.

4. Защитные проверки: деление и короткое замыкание

Деление на ноль — классическая проблема. И особенно обидно, что она часто появляется не потому, что вы хотели делить на ноль, а потому что вы не подумали, что пользователь может ввести 0. А пользователь может. Даже если вы просили «введите делитель» — он всё равно может.

Короткое замыкание в && — это тот редкий случай, когда «ленивость» оператора делает вашу программу безопаснее: если левая часть ложна, правая даже не вычисляется. Значит, если слева стоит проверка безопасности, справа можно ставить опасное действие.

Правильный порядок частей в &&

#include 

int main() {
  int a = 0;
  int b = 0;
  std::cin >> a >> b;

  if (b != 0 && (a / b > 2)) {
    std::cout << "a/b > 2\n";
  } else {
    std::cout << "condition is false\n";
  }
}

Если b == 0, левая часть b != 0 ложна, и деления не будет. Это и есть защитная проверка.

Неправильный порядок (опасно)

#include 

int main() {
  int a = 0;
  int b = 0;
  std::cin >> a >> b;

  if ((a / b > 2) && b != 0) {
    // ОПАСНО: деление уже произошло
    std::cout << "a/b > 2\n";
  }
}

Даже если вы «проверяете b», вы проверяете его слишком поздно. Это как надеть шлем после падения с велосипеда: шлем отличный, но тайминг подкачал.

5. Практический мини‑пример: одна команда — одно действие

Сейчас соберём всё в маленькое приложение, которое показывает стиль «надёжных условий». Мы не используем циклы и функции (кроме main), потому что их ещё не проходили. Но мы уже можем сделать программу, которая не падает от каждого неудачного ввода и не делает опасных операций без проверок.

Идея: пользователь вводит команду (divide, first, last), затем необходимые данные.

Каркас ветвления по команде

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  if (cmd == "divide") {
    std::cout << "mode: divide\n";
  } else if (cmd == "first") {
    std::cout << "mode: first\n";
  } else {
    std::cout << "unknown command\n";
  }
}

Это базовая «развилка»: команда — это строка, сравнение строк даёт bool, if/else if/else выбирает одну ветку.

Команда divide: проверяем ввод и защищаем деление

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  if (cmd == "divide") {
    int a = 0, b = 0;
    if (std::cin >> a >> b) {
      if (b != 0) std::cout << (a / b) << '\n';
      else std::cout << "division by zero\n";
    } else {
      std::cout << "input error\n";
    }
  }
}

Тут сразу две линии защиты: сначала мы убеждаемся, что числа вообще прочитались, потом убеждаемся, что делитель не равен нулю.

Команда first: читаем строку и проверяем пустоту

С first есть нюанс: после std::cin >> cmd в буфере может остаться перевод строки, и getline может прочитать пустую строку. Мы подробно говорили об этом в теме про getline, поэтому здесь просто применяем аккуратный приём: читать строку через std::getline(std::cin >> std::ws, s) — std::ws «съест» пробелы и переводы строк перед реальным текстом.

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  if (cmd == "first") {
    std::string s;
    std::getline(std::cin >> std::ws, s);

    if (s.size() == 0) std::cout << "empty\n";
    else std::cout << s[0] << '\n';
  }
}

Да, это чуть «магически» выглядит из-за std::ws, но вы уже встречали проблему «пустой строки после >>», и это одно из стандартных решений.

Команда last: последний символ без выхода за границы

#include 
#include 

int main() {
  std::string cmd;
  std::cin >> cmd;

  if (cmd == "last") {
    std::string s;
    std::getline(std::cin >> std::ws, s);

    if (s.size() > 0) std::cout << s[s.size() - 1] << '\n';
    else std::cout << "empty\n";
  }
}

Смысл тот же: сначала проверка, потом индексирование.

6. Типичные ошибки

Ошибка №1: = вместо == в условии.

Это одна из самых неприятных ошибок, потому что она часто не мешает компиляции. = меняет переменную и возвращает присвоенное значение, поэтому условие может стать «ложным» или «истинным» не из-за сравнения, а из-за того, что вы только что что-то перезаписали. Лечится привычкой внимательно смотреть на «центральный оператор» условия и писать сравнение явно: x == 0, cmd == "exit".

Ошибка №2: перепутанная граница (> вместо >=, < вместо <=).

Такие ошибки почти всегда проявляются «ровно на одном значении»: 18, 0, «последний индекс». Поэтому они живут долго: вы тестируете 20, 30, 40 — всё работает. Привычка, которая помогает, звучит по‑детски, но работает железно: сначала сказать условие словами («включая 18»), потом выбрать знак.

Ошибка №3: индексирование строки без проверки длины.

std::string не обязан быть непустым. Если вы делаете s[0] или s[s.size() - 1], сначала должно быть условие s.size() > 0. Особенно коварен «последний символ», потому что выражение s.size() - 1 выглядит безобидно, но на пустой строке превращается в неправильный индекс. Защита — отдельная ветка if или левый аргумент в &&.

Ошибка №4: деление без проверки нуля (или проверка стоит «справа»).

Проверка b != 0 должна стоять до деления. Самый удобный паттерн — b != 0 && (a / b > ...), где короткое замыкание гарантирует, что деление не произойдёт при b == 0. Если поменять местами части условия, вы получите опасное поведение даже при наличии «проверки».

Ошибка №5: отсутствие минимальной проверки ввода.

Если вы делаете std::cin >> x; и сразу используете x, вы предполагаете корректный ввод. На практике это приводит к странным «залипаниям» и некорректным результатам, когда пользователь вводит не число. Даже без углубления в устройство потоков уже можно писать надёжнее: if (std::cin >> x) { ... } else { ... }. Это добавляет пару строк, но экономит часы удивления.

3. Самостоятельные курсы JavaRush: свобода и интерактив

Для тех, кто любит учиться в своём ритме, JavaRush предлагает самостоятельные курсы по подписке. Это оптимальный выбор для занятых людей, студентов и тех, кто хочет экономить деньги

Как всё устроено

После покупки подписки вы получаете доступ ко всем урокам выбранного курса. Стартовать можно с любого уровня — хоть с полного нуля, хоть с продвинутых тем, если опыт уже есть. Курс состоит из коротких теоретических блоков и большого количества практических заданий: по каждому разделу вас ждёт десятки и сотни упражнений

Формат обучения геймифицирован: за успешные задания вы получаете очки, открываете новые уровни, двигаетесь по “квесту” и видите свой прогресс. Приятный бонус — автоматическая проверка кода, которая сразу показывает, где вы ошиблись, и даёт рекомендации по исправлению. Всё обучение проходит онлайн, так что вы можете учиться с ноутбука, телефона или даже в специальном плагине для IDE

Если возникают сложности, работает живой форум. Здесь можно задать вопрос по любой задаче и быстро получить ответ от сообщества или опытных студентов. Многим форум помогал пройти сложные моменты, когда не было возможности спросить у преподавателя 🙋‍♂️ Такая система поддержки и геймификация делают самостоятельное обучение максимально эффективным.

4. Самостоятельно или с ментором: как выбрать?

У каждого формата есть свои плюсы. Давайте коротко сравним их:

  • Глубина и содержание: в университете программа шире и глубже, вы проходите не только язык, но и современные технологии, учитесь работать в команде, решаете реальные проекты и готовитесь к собеседованиям. В самостоятельном формате упор на алгоритмы, язык и индивидуальную практику.
  • Формат обучения: самостоятельно — это максимум свободы, без расписания, вы сами отвечаете за прогресс. С ментором — занятия по расписанию, постоянная поддержка, команда, групповые проекты.
  • Поддержка и обратная связь: самостоятельные курсы — автоматическая проверка, форум, советы от других студентов. Университет — персональный наставник, чат, регулярная связь, консультации HR.
  • Стоимость: самостоятельное обучение дешевле, университет — дороже, но с большим количеством персональной работы и карьерной поддержкой.
  • Мотивация и дисциплина: если нужна внешняя мотивация, строгий ритм и ощущение “университета” — лучше выбрать формат с ментором. Если вы самодисциплинированы и любите свободу — самостоятельные курсы вас устроят.
  • Результаты: оба пути ведут к знаниям и старту карьеры. Но после университета у вас будет больше опыта в командной работе, проектов для портфолио и готовых резюме — работодатели это очень ценят.
